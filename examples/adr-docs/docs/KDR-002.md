# KDR 002: Adoption of a Canonical Data Model (CDM) for Enterprise Integration

**Status:**

**Proposed:**

## Context

The current data landscape within Howden is fragmented increasing the complexity of
integrations significantly. Different systems (CRM, General Ledger, PAS etc.,) use varying
definitions for core entities like "Opportunity" or "Trial Balance"

- **The Problem:** We are currently performing N * N custom mappings for every new integration,
leading to "spaghetti" logic and high maintenance costs.

- **The Goal:** Create a single, platform-agnostic "Golden Schema" that all systems must map to
when exchanging data.

The Canonical Data Model(CDM) Governance Process (TDA Integration)
Because a CDM change can break all integrations to it, it follows a stricter version of the TDA
process:

- **Draft:** Data Architect proposes a field change in the CDM SharePoint List.
- **Impact Analysis:** Automation identifies which "Accepted" KDRs or existing systems use that
entity.
- **TDA Tabling:** The change must be presented at the TDA Forum to verify that downstream
consumers (Analytics, Finance) can handle the change.
- **Accepted:** The CDM JSON Schema is updated and versioned (e.g., /v1/customer ->
/v2/customer).

## Decision

A **Canonical Data Model (CDM)** defined and owned by the Digital Backbone will serve as the
universal language for all asynchronous and synchronous data exchanges through the Digital
Backbone.

### Implementation for Users

- **Decoupled Producers:** Data producers (e.g., Salesforce) will transform their internal data
structures into the CDM format before publishing to the Service Bus/Event Hub.

- **Simplified Consumption:** Data consumers (e.g., Data Warehouse, PAS, Payroll) only need to
understand the CDM schema, regardless of the source system.
- **Standardisation:** We will use JSON Schema for definition
- **Governance:** Any changes to the CDM must be tabled at the TDA Forum via the SharePoint
KDR process to ensure backward compatibility.

## Consequences

### Pros

- **Reduced Mapping Complexity:** New systems only need one mapping (Source → CDM)
instead of mapping to every other existing system.

- **Data Consistency:** Definitions for fields like ISO_Currency_Code or
Global_Customer_ID are enforced globally.
- **Vendor Agnostic:** If we replace our CRM, the consumers of "Customer" data are unaffected
because the CDM contract remains the same.

### Cons

- **Initial Overhead:** The first few integrations will take longer as we iterate on the canonical model definition for the CDM.

- **Transformation Latency:** Every message requires a transformation step (Mapping), which
adds a few milliseconds of overhead.

- **Lowest Common Denominator:** There is a risk the CDM becomes too generic to be useful, or
too bloated by trying to include every field from every system.
